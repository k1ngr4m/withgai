## 1. 设计目标

### 1.1 一句话目标
用短促、清晰、带办公系统质感的动效，让玩家在后端 MVP 中能立即读懂“我打出了什么、谁受到了影响、资源发生了什么变化、下一步要做什么”。

### 1.2 动效体验关键词
- `职场后台感`
  动效像企业后台、监控面板、服务告警、工单流转和楼宇广播，而不是魔法特效或中世纪战斗演出。
- `冷白办公楼`
  以冷白、灰蓝、青绿技术光、系统橙、警示红为主。高亮只在关键反馈出现，避免全屏长期闪烁。
- `短促可读`
  大多数反馈控制在 `0.10-0.45s`。玩家连续出牌时，动效不能阻塞操作。
- `构筑反馈优先`
  后端职业的服务、缓存、请求、防线和长期状态必须比普通 UI 装饰更醒目。
- `MVP 可落地`
  优先用 `Tween`、`modulate`、`scale`、`position`、短时 `Label` 飘字和 overlay 实现，不要求先重构 UI 架构。

### 1.3 非目标
- 不做五职业完整战斗动效。
- 不做复杂时间轴 cutscene。
- 不做大规模角色 sprite 战斗演出重构。
- 不做长时间遮挡 UI 的华丽全屏演出。
- 不把动效做成必须等待完成才允许玩家继续操作的硬阻塞流程，除非是胜负、进出场景等流程节点。

## 2. 全局动效规范

### 2.1 时间与节奏
| 类型 | 建议时长 | 用途 |
| --- | --- | --- |
| 微交互 | `0.08-0.16s` | 按钮按下、卡牌点击、节点 hover / focus |
| 普通反馈 | `0.18-0.32s` | 伤害、防线、资源变化、选择确认 |
| 重点反馈 | `0.35-0.55s` | 打出能力牌、Boss 进入、胜利奖励出现 |
| 场景过渡 | `0.25-0.50s` | 地图进入战斗、战斗进入奖励、返回主菜单 |

### 2.2 缓动建议
- 入场：`Tween.TRANS_QUAD` + `Tween.EASE_OUT`
- 退出：`Tween.TRANS_QUAD` + `Tween.EASE_IN`
- 冲击：`Tween.TRANS_BACK` + `Tween.EASE_OUT`，只用于卡牌命中、资源爆发、结算数字。
- 循环氛围：使用 `_process(delta)` 的低幅度 `sin` 变化，参考当前主菜单环境线条和 pulse 节点。

### 2.3 位移、缩放与透明度
- UI 入场位移：`12-28px`，从下方或右侧轻入场。
- 卡牌按压缩放：`0.96 -> 1.00`，不要超过 `1.06`。
- 命中抖动：敌方头像或面板 `6-10px` 横向微抖，持续不超过 `0.18s`。
- 高亮呼吸：透明度 `0.72-1.00` 或描边 alpha 变化，避免大幅缩放。
- 飘字：从目标上方偏移 `-28px` 到 `-54px`，同时 alpha `1.0 -> 0.0`。

### 2.4 色彩语义
| 语义 | 主色建议 | 使用位置 |
| --- | --- | --- |
| 伤害 / 压力 | 警示红 `#ff5a5f` | 敌我 HP 变化、受到攻击 |
| 防线 / 稳定 | 冷青 `#72d6e8` | 获得防线、熔断保护 |
| 缓存 / 资源积累 | 荧光绿 `#65f0a2` | 缓存增加、Redis 预热 |
| 服务在线 | 技术青 `#5bd6ff` | 服务上线、API 网关 |
| 请求队列 | 系统橙 `#ffb454` | 请求堆积、回合末清算 |
| 失败 / 危险 | 深红 + 暗化 | 精神过低、失败结算 |
| 奖励 / 收益 | 暖金 `#ffe36e` | 绩效点、窝囊费、遗物 |

### 2.5 可访问性与性能
- 关键状态不能只靠颜色表达，必须配合文本、数字或图标变化。
- 所有循环氛围动效都应低 alpha、低频率，避免造成视觉疲劳。
- 低性能模式可关闭背景呼吸线、粒子和重复 pulse，但保留出牌、伤害、资源变化等核心反馈。
- 不要求屏幕震动作为 P0；如后续加入，必须可关闭。

## 3. 场景级动效

### 3.1 主菜单 `MainMenuScene`
当前主菜单已有背景、广播条、环境线条 pulse、职业 spotlight 轮播。MVP 只在现有基础上统一节奏，不重写布局。

P0 必做：
- 进入主菜单时，标题、广播条、职业 spotlight、右侧主按钮按 `0.05s` 间隔依次淡入。
- `开始爬楼`、`继续`、`工位成长`按钮按下时做 `0.10s` 按压缩放，并在切场前出现短促冷青描边闪烁。
- 可继续存档存在时，`ContinueButton` 轻微呼吸；无存档时保持静态禁用。
- 当前“后端首个可玩”状态 chip 保持微弱 pulse，用于强调 MVP 范围。

P1 增强：
- 楼宇广播文本切换时做横向扫入，而不是直接替换。
- 后端职业 spotlight 切换到可出战态时，角色图有一次冷青扫描线。

### 3.2 职业选择 `ClassSelectScene`
职业选择页承担“后端可玩，其他职业暂锁定”的范围说明。

P0 必做：
- 职业卡片入场按从左到右 `0.04s` 递进淡入。
- 后端卡片有稳定冷青边框呼吸，表示可出战。
- 锁定职业卡片整体饱和度低、透明度略低，按钮按下无跳转但有 `0.12s` 禁止反馈抖动。
- 点击后端开始时，卡片缩放到 `1.03` 后回落，并进入地图。

P1 增强：
- 锁定职业显示灰色“门禁未开”扫描线，提示其为未来内容，而非错误状态。

### 3.3 地图 `MapScene`
地图是爬楼选择，不需要大地图演出，重点是路线可读。

P0 必做：
- 可选节点做轻微冷青外框 pulse。
- 已完成节点淡出为灰蓝低亮，并显示“已完成”文本。
- 点击节点时，按钮先做 `0.12s` 按压，再出现一个从当前节点向场景边缘扩散的短线条，随后切到目标场景。
- Boss 节点使用警示红或系统橙短闪，不做长时间闪烁。

P1 增强：
- 当前楼层列轻微上浮 `4px`，提示当前进度。
- 地图背景增加低 alpha 电梯楼层指示灯流动。

### 3.4 战斗 `BattleScene`
战斗是 MVP 动效重点。当前 UI 以 `_build()` 重建为主，因此动效应优先做成短时 overlay 或节点局部 Tween，避免阻塞战斗服务逻辑。

P0 必做：
- 打出卡牌：
  - 被点击的卡牌按钮按压缩放 `0.96 -> 1.00`。
  - 成功打出后，在手牌区域生成短暂残影或冷青扫线，表示操作被接收。
  - 不能打出的卡牌做 `0.12s` 横向轻抖，并保持禁用态。
- 伤害：
  - 目标敌方面板短红闪 + 轻微横向抖动。
  - 敌方上方出现红色飘字，例如 `-8 精神` 或 `-8`。
- 防线：
  - 玩家资源行或防线数字出现冷青 pulse。
  - 防线增加时出现 `+X 防线` 飘字。
- 精力：
  - 精力消耗时数字快速下沉 `6px` 并恢复。
  - 精力不足时卡牌费用文本变橙红闪一次。
- 敌方意图：
  - 选中目标的 `▶` 前缀保持；选中切换时，目标面板外框做一次冷青闪烁。
  - 意图被降低时，意图文本先橙色闪烁，再更新为新数值。
- 回合切换：
  - 点击结束回合后，玩家区暗化 `0.20s`，敌人行动时敌方面板依次亮起。
  - 新玩家回合开始时，资源行做一次冷白扫描。
- 胜负：
  - 胜利时敌方面板整体淡出，奖励入口出现暖金闪。
  - 失败时画面整体降低亮度，精神数值红色闪烁后进入结算。

P1 增强：
- 根据卡牌类型区分动效：
  - `attack`：红色命中线或工牌甩击感。
  - `skill`：冷青防护框、缓存粒点。
  - `power`：持续状态图标入场并停留 pulse。
- 战斗日志新增短暂高亮最新一行，帮助玩家追踪结算。

### 3.5 奖励 `RewardScene`
奖励页要给玩家“战斗结束、可以拿收益”的轻松感。

P0 必做：
- 页面入场时标题和绩效点数字暖金淡入。
- 三张候选卡按顺序弹入，缩放 `0.94 -> 1.00`。
- 选中卡牌时，卡牌边框暖金高亮，`✓` 标记同步出现。
- 跳过卡牌时使用低亮灰蓝反馈，不使用红色，避免像错误。
- 确认领取时，已选卡牌向玩家牌组方向轻微位移后淡出，再切回地图。

P1 增强：
- 如果出现遗物候选，遗物按钮用一次暖金光圈表示稀有收益。

### 3.6 商店 `ShopScene`
商店动效应像自动贩卖机和深夜工位采购系统。

P0 必做：
- 商品列表入场时按顺序淡入。
- 买卡 / 买遗物成功时，商品按钮短暂暖金闪烁后禁用或消失。
- 绩效点减少时，绩效点数字做橙色下跳反馈。
- 金额不足时，按钮轻抖并闪红，不弹全屏提示。
- 刷新商品时，商品行先横向扫出，再新商品扫入。
- 移除卡牌成功时，所选牌按钮灰化并出现 `已移除` 反馈。

P1 增强：
- 商店背景可有低频售货机灯光 pulse，但不得影响商品可读性。

### 3.7 随机事件 `EventScene`
事件页应有文字阅读节奏，不需要过多动效。

P0 必做：
- 事件标题先入场，正文延迟 `0.08s` 淡入，选项延迟 `0.12s` 逐个出现。
- 选择选项后，选中按钮冷青高亮，其他选项淡出，再回地图。
- 事件造成收益或损失时，使用对应颜色飘字；如果当前实现没有结果页，至少在切场前保留 `0.20s` 选择确认反馈。

P1 增强：
- 事件背景可轻微暗化，让正文面板更聚焦。

### 3.8 休息处 `RestScene`
休息处是少数安全场景，动效应该更慢、更柔和。

P0 必做：
- 进入休息处时背景不闪烁，按钮轻柔淡入。
- 选择冥想恢复时，精神数值暖青色递增反馈，随后回地图。
- 选择复盘升级时，卡牌列表淡入；升级成功的牌出现暖金 `+` 闪烁。

P1 增强：
- 茶水间背景可以有低频暖光呼吸，与战斗冷光区分。

### 3.9 结算 `RunResultScene`
结算页要清楚表达本局结果与窝囊费收益。

P0 必做：
- 胜利标题使用暖金入场；失败标题使用低亮白 + 红色短闪。
- 最高楼层、击败 Boss、窝囊费三项数字逐项出现。
- 窝囊费数字使用暖金跳字，强调局外成长收益。
- 返回主菜单和查看工位成长按钮在统计出现后再淡入。

P1 增强：
- 失败时背景轻微暗化，胜利时保持高亮但不过曝。

## 4. 后端职业专项动效

### 4.1 后端动效语言
后端职业的动效应像“系统稳定运行”而不是“爆炸输出”。视觉重点是服务节点上线、缓存层叠、队列堆积、网关转发和无损发布。

通用元素：
- 小型服务节点：冷青圆点或方块，围绕玩家资源行短暂出现。
- 缓存层：绿色矩形栈或水平条，随数值增加叠层。
- 请求队列：橙色小点排队进入队列，回合末转化为伤害。
- 网关扫描：冷青横向扫描线，用于能力牌和长期状态。

### 4.2 `service_online` 服务在线
触发场景：服务上线、回合开始 / 回合结束服务收益。

P0 表现：
- 获得服务时，玩家资源区出现 `+1 服务` 冷青飘字。
- 服务数值变化时，资源文本或图标做 `0.18s` 冷青 pulse。
- 回合开始由服务产生收益时，从服务文本向防线 / 缓存文本方向出现短线流动。

验收：
- 玩家能在不读日志的情况下感知“服务上线了”。

### 4.3 `cache` 缓存
触发场景：缓存命中、Redis 预热、分库分表额外缓存、全量回写消耗缓存。

P0 表现：
- 增加缓存时显示 `+X 缓存` 绿色飘字。
- 缓存被消耗时，缓存数字先绿色闪烁，再快速收缩回常态。
- 缓存参与伤害时，从缓存数字向目标敌人发出短绿色线条。

验收：
- 玩家能区分“获得缓存”和“消耗缓存造成伤害”。

### 4.4 `api_gateway` API 网关
触发卡牌：`card_backend_api_gateway`

P0 表现：
- 打出能力牌时，玩家区域上方出现一次冷青网关扫描线。
- 状态入场时显示 `API网关上线`。
- 回合开始触发防线时，从 `API网关` 状态位置弹出 `+X 防线`。
- 若服务不少于 2 个并触发抽牌，手牌区域出现一次冷青扫入。

验收：
- 玩家能看出这是长期能力，而不是一次性技能。

### 4.5 `redis_warmup` Redis 预热
触发卡牌：`card_backend_redis_warmup`

P0 表现：
- 打出时缓存获得使用绿色大号飘字，优先级高于普通缓存。
- 下回合首张牌费用降低时，目标卡牌费用文本冷绿闪烁，并短暂显示 `预热 -1`。
- 折扣消耗后，费用文本回到普通颜色。

验收：
- 玩家能在费用变化时理解来源是 Redis 预热，而不是显示错误。

### 4.6 `request_queue` 请求队列
触发卡牌：`card_backend_message_queue`

P0 表现：
- 生成请求时显示橙色 `+X 请求`。
- 回合末请求结算时，请求数值向所有敌人分裂成橙色短线。
- 每个受影响敌人显示对应伤害飘字。
- 请求消耗后，请求资源文本做一次收缩淡出。

验收：
- 玩家能看出请求队列是延迟到回合末清算的资源。

### 4.7 `sharding` 分库分表
触发卡牌：`card_backend_sharding`

P0 表现：
- 打出能力牌时，玩家资源区出现多层绿色小栈。
- 每回合第一次获得缓存并触发额外缓存时，额外的 `+1 缓存` 使用更亮的绿色，并附带 `分库分表` 小标签。
- 同一回合后续缓存不重复显示该标签。

验收：
- 玩家能理解这是“每回合第一次缓存收益增强”。

### 4.8 `card_backend_flush_all` 全量回写
触发卡牌：`card_backend_flush_all`

P0 表现：
- 出牌时缓存数字先高亮，随后快速归零或降低。
- 从玩家资源区向目标敌人发出绿色到红色的短线转换。
- 目标敌人出现更大的伤害飘字，颜色为红色，外圈带一点绿色残影。

验收：
- 玩家能明确看到“缓存被花掉，换成高额伤害”。

### 4.9 防守与意图控制
相关卡牌：`card_backend_circuit_breaker`、`card_backend_service_degrade`、`card_backend_traffic_shaping`

P0 表现：
- `熔断保护`：防线获得时使用冷青护框；若因服务或高攻击意图增强，额外显示 `熔断加固`。
- `服务降级`：所有敌方攻击意图文本橙色闪烁并降低，敌方面板不做伤害抖动。
- `流量削峰`：根据本回合承受压力转换缓存时，先红色压力小点汇入玩家，再变成绿色缓存。

验收：
- 玩家能区分“造成伤害”和“削弱意图 / 转换压力”。

## 5. MVP 优先级

### 5.1 P0 必做
- 按钮 / 卡牌点击的基础按压反馈。
- 战斗伤害、防线、精力、缓存、服务、请求变化的飘字和颜色反馈。
- 选目标、敌方意图变化、结束回合、新回合开始反馈。
- 地图可选节点 pulse 与进入节点反馈。
- 奖励选择、商店购买、休息恢复、结算数字入场。
- 主菜单现有氛围动效保留，并统一按钮点击反馈。

### 5.2 P1 增强
- 卡牌类型差异化动效。
- 后端长期能力状态图标入场和回合触发线条。
- 商店刷新扫入扫出。
- 事件选择结果短反馈。
- 战斗日志最新行高亮。

### 5.3 P2 后续
- 五职业完整动效包。
- 角色 sprite idle / cast / hurt 接入战斗。
- 更完整的粒子系统和屏幕震动设置。
- Boss 专属入场与阶段演出。
- 动效设置页：减少动效、关闭屏幕震动、关闭背景氛围。

## 6. Godot 落地建议

### 6.1 推荐实现路径
当前 UI 多数场景在操作后调用 `_build()` 重建界面。MVP 实现应减少侵入：

1. 在 `Scripts/UI/UiFactory.gd` 中补充轻量动效 helper，或新建 `Scripts/UI/UiMotion.gd`。
2. 场景内操作成功后，先播放局部短反馈，再调用原有业务方法和 `_build()`。
3. 对 `_build()` 后才知道的新状态，使用一次性入场动效，例如新生成的卡牌按钮、敌人面板、奖励按钮。
4. 战斗数值变化需要对比前后状态时，可在 `BattleScene` 保存上一帧轻量快照，只记录 HP、防线、精力、后端资源和敌方意图，不作为业务真状态。

### 6.2 建议通用方法
可放在 `UiMotion` 或 `UiFactory`：

```gdscript
static func pulse(node: CanvasItem, color: Color, duration := 0.18) -> void
static func press(node: Control, duration := 0.10) -> void
static func shake(node: Control, amount := 8.0, duration := 0.14) -> void
static func float_text(parent: Control, text: String, color: Color, from: Vector2) -> void
static func fade_in(node: CanvasItem, duration := 0.20, offset := Vector2.ZERO) -> void
static func flash_modulate(node: CanvasItem, color: Color, duration := 0.16) -> void
```

### 6.3 实现约束
- 动效 helper 不读取或修改 `RunSession`、`BattleService`、`RewardService` 的业务状态。
- 表现层快照只用于比较显示差异，不能作为真状态。
- 不在配置表中加入动效字段作为 MVP 必需项；如后续需要，可再增加独立表现配置。
- 所有动效失败时不应影响业务流程。
- 场景切换前的短反馈总时长不超过 `0.25s`，避免拖慢 MVP 体验。

### 6.4 与现有主菜单兼容
- 保留 `MainMenuScene` 中已有 `_process` 氛围线条、broadcast 轮换、spotlight 轮播。
- 新增按钮反馈时避免重复创建长期 Tween；点击反馈只在事件发生时创建。
- 如果页面 rebuild，先清理旧引用数组，避免已释放节点继续被 `_process` 访问。当前主菜单已有 `is_instance_valid` 检查，应继续沿用。

## 7. 验收标准

### 7.1 文档验收
- 本文件位于 `Resources/Docs/05_MOTION_DESIGN_MVP.md`。
- 文档明确以后端公开可玩 MVP 为范围。
- 每个核心场景都有 P0 动效要求。
- 后端专项覆盖：
  - `service_online`
  - `cache`
  - `api_gateway`
  - `redis_warmup`
  - `request_queue`
  - `sharding`
  - `card_backend_flush_all`
- 文档包含 Godot 4 / GDScript 落地建议，并说明不要求先重构业务状态层。

### 7.2 后续实现验收
- 出牌时卡牌有按压或成功反馈。
- 伤害、防线、缓存、服务、请求变化都有可见反馈。
- 敌人攻击、意图降低、胜利、失败、奖励选择、商店购买、地图进入节点都有短动效。
- 主菜单现有氛围动效继续保留，不与新增规范冲突。
- 快速连续操作不会因为动效阻塞而明显变慢。
- 关闭或弱化氛围动效后，核心战斗反馈仍可读。
